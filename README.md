# Benchmarking Quantum vs Classical Approaches for Time-Dependent TSP

Companion repository for the IEEE Quantum Week 2026 (IEEE QCE) paper
*Benchmarking Quantum vs Classical Approaches for Time-Dependent TSP* by
M. A. Khan, K. Ganesh, and S. Hossain.

The repository contains:

- the autonomous, single-file solver code generated by the SuperQ "Super"
  optimization platform for each backend (no clustering),
- the human-refined cluster-based solvers used to scale the benchmark to 100
  cities,
- the deterministic problem instances used in the paper,
- the result JSONs the paper reports on,
- the LaTeX source for the paper.

## Solvers

Four optimization paradigms are evaluated on the same instances:

| Solver        | Type                            | Platform                              |
|---------------|---------------------------------|---------------------------------------|
| Gurobi        | Classical MILP (MTZ)            | Local (WLS license)                   |
| D-Wave        | Quantum annealing (CQM)         | D-Wave Leap cloud                     |
| QAOA          | Gate-based quantum (QUBO)       | AWS Braket SV1, Rigetti Cepheus-1-108Q|
| Quanfluence   | Quantum-inspired Ising machine  | Quanfluence REST API                  |

### Autonomous vs refined

`code/autonomous/` holds the single-file solvers as produced by the SuperQ
"Super" autonomous optimization platform. Each script solves a TSP instance
end-to-end on its respective backend without clustering and is intended as a
direct artefact of the platform output that accompanies the paper.

`code/refined/solvers/` contains the human-refined pipeline. All four backends
share the same cluster decomposition strategy:

1. Estimate 2D coordinates from the input distance matrix via MDS.
2. Partition the cities into clusters using K-means, sized to each backend's
   hardware limit.
3. Solve each cluster's TSP on the target backend.
4. Stitch the cluster tours together using a greedy nearest-neighbour
   connection.
5. Apply 2-opt local search on the full stitched tour.

Per-backend cluster size limits used in the paper:

| Backend     | Max cluster size | Notes                                      |
|-------------|------------------|--------------------------------------------|
| Gurobi      | 15               | Limited by per-cluster solve time.         |
| D-Wave      | 15               | Limited by hybrid CQM submission size.     |
| QAOA (SV1)  | 5  (25 qubits)   | Simulator memory and runtime.              |
| QAOA (QPU)  | 4  (16 qubits)   | Rigetti Cepheus 20,000-gate compile limit. |
| Quanfluence | 10 (100 vars)    | API throughput.                            |

## Installation

Python 3.10+ is required.

```bash
pip install -r requirements.txt
```

Backend-specific prerequisites:

- Gurobi 13.0+ with a valid license (academic WLS works; place at
  `gurobi.lic` and point `GRB_LICENSE_FILE` at it).
- AWS account with Amazon Braket access for QAOA runs.
- D-Wave Leap account and API token.
- Quanfluence API credentials.

Credentials are read from a `.env` file at the repository root:

```env
AWS_ACCESS_KEY_ID=...
AWS_SECRET_ACCESS_KEY=...
AWS_DEFAULT_REGION=us-east-1

DWAVE_API_TOKEN=...

QUANFLUENCE_BASE_URL=https://gateway.quanfluence.com
QUANFLUENCE_USERNAME=...
QUANFLUENCE_PASSWORD=...
QUANFLUENCE_DEVICE_ID=41

GRB_LICENSE_FILE=gurobi.lic
```

## Reproducing the paper

### 1. Generate the benchmark instances

```bash
python data/generate_instances.py
```

This writes deterministic Euclidean TSP instances for `n in {5, 10, 25, 50,
100}` to `data/instances/`. Every instance is generated from the seed `42` and
contains the city labels, 2D coordinates, and the rounded NxN distance matrix.

### 2. Run the cluster benchmarks

From the repository root:

```bash
python code/refined/benchmarks/bench_gurobi.py
python code/refined/benchmarks/bench_dwave.py
python code/refined/benchmarks/bench_quanfluence.py
python code/refined/benchmarks/bench_qaoa.py
python code/refined/benchmarks/bench_qaoa_rigetti.py
```

Each script loads the instances from `data/instances/`, runs the matching
cluster solver across all five sizes, and writes its results to
`results/results_<solver>.json`.

### Auxiliary scripts

```bash
python code/refined/benchmarks/check_braket_devices.py
python code/refined/benchmarks/estimate_gate_counts.py
```

`check_braket_devices.py` lists available AWS Braket QPUs and simulators.
`estimate_gate_counts.py` reports the projected gate count of the QAOA TSP
circuit as a function of city count and QAOA depth, and identifies the
maximum cluster size that fits in the Rigetti Cepheus-1-108Q gate budget.

## Result format

Each entry in a results file looks like:

```json
{
  "n": 100,
  "distance": 781.56,
  "raw_distance": 963.31,
  "n_clusters": 7,
  "time": 151.85,
  "status": "optimal",
  "solver": "Cluster-D-Wave"
}
```

`raw_distance` is the stitched tour length before 2-opt; `distance` is the
final tour length after 2-opt refinement.
